# Miyu 设计方案

本文记录 Miyu 从 AstrBot 插件体系迁移为独立 CLI AI Agent 的设计约定。设计会随实现推进更新，未确认内容只作为候选方案，不视为最终承诺。

## 目标定位

Miyu 是一个轻量、快速、单二进制发布的 CLI AI Agent。

- 使用 Rust 实现核心功能，仅发布编译后的二进制文件。
- 默认不依赖 Python、Node、浏览器、NapCat 或 AstrBot 运行时。
- 保留 Miyu 的人格、提示词、skills、插件能力和本地知识库方向。
- 内置提示词编译进二进制，用户不能直接修改内置提示词，但可以通过配置覆盖或追加。
- 支持模块化 skills 和插件系统，内置插件默认随二进制发布。
- 项目不是开源产品，源码、提示词和插件逻辑不作为公开交付物。

需要注意：单二进制发布只能提高逆向成本，不能保证提示词和逻辑绝对不可提取。敏感提示词应尽量最小化暴露，并在必要时做分段、混淆或远端策略。

## 第一版范围

第一版先确认 CLI、配置、路径、Provider 和无插件对话能力，不迁移具体 AstrBot 插件。

第一版包含：

- `miyu [MESSAGE...]`：主对话入口。
- `miyu`：无参数进入轻量 REPL。
- `miyu ask [MESSAGE...]`：兼容别名，不作为主入口。
- `miyu init`：只创建 XDG 目录、默认配置和状态文件，不进入配置界面。
- `miyu config validate`：校验 JSONC 配置。
- `miyu config paths` 或 `miyu paths`：显示实际路径。
- `miyu config`：进入类似 `rime-llm-translator` 的 TUI 配置界面，用于配置 Provider、模型和全局参数。
- `miyu fish-init`：安装 fish unknown command hook，创建或覆盖 `~/.config/fish/conf.d/miyu.fish`。
- `miyu history`：渲染当前会话历史，默认显示最近 20 条。
- `miyu new`：开始新会话，把当前 `conversation.jsonl` 归档为带时间戳的历史文件。
- `miyu reset`：清空当前会话历史，不归档。
- 空工具注册表和插件 trait，为后续迁移插件预留。

第一版不包含：

- 具体插件迁移。
- 完整对话 TUI。
- 自动安装 zsh/bash shell hook。
- 图片预览。
- 知识库索引。

## CLI 语义

Miyu 的主入口应尽量接近日常终端输入习惯。

```sh
miyu 你好
miyu 帮我 写一个 rust cli
miyu "帮我总结一下 Rust trait"
```

规则：

- 所有位置参数会用空格拼成一条用户消息。
- 普通中文和简单自然语言不需要引号。
- 只有包含 shell 特殊字符、管道、变量展开、通配符、换行或需要保留连续空格时，才需要引号。
- 后续支持 `miyu -` 从 stdin 读取长文本。

无参数 `miyu` 默认进入 REPL，而不是打印帮助。帮助信息仍可通过 `miyu --help` 查看。

## Shell 拦截体验

Miyu 希望像一个住在终端里的助手：用户直接在 shell 输入自然语言，如果该输入不是系统命令，shell 的 `command not found` hook 可以把它交给 Miyu。

目标体验：

```fish
> 你好
Miyu: 你好呀。你这是把终端当聊天框用了是吧。
```

候选实现：

- Fish：接管 `fish_command_not_found`。
- Zsh：接管 `command_not_found_handler`。
- Bash：接管 `command_not_found_handle`，但兼容性较差。

第一阶段只实现：

- `miyu --shell-intercept --shell fish -- <argv...>`：专用拦截入口。
- `miyu fish-init`：安装 fish hook，创建或覆盖 `~/.config/fish/conf.d/miyu.fish`。

第二阶段再实现：

- `miyu zsh-init` / `miyu bash-init`：后续再考虑。
- 类似 tash 的命令行 buffer 回填模式。

默认行为：

- 只拦截看起来像自然语言的 unknown command。
- 如果像拼错命令，例如 `gti status`，优先提示可能是 `git status`。
- 如果是普通聊天，直接回复。
- 如果是命令建议，只展示命令，不自动执行。
- 对危险命令建议做本地规则标记和二次确认。

第一版不把 shell 拦截开关暴露为全局配置项。是否安装 fish hook 由 `miyu fish-init` 决定；自然语言判定和安全策略先由代码内置实现。Miyu 的语气、吐槽和人格表现由提示词本身控制。

## 配置格式

主配置使用 JSONC。

原因：

- 保留 JSON 生态、schema 校验和工具兼容性。
- 支持注释和尾逗号，适合人工维护。
- 比 TOML 更适合复杂嵌套 Provider、插件和 skills 配置。
- 比 YAML 更少隐式类型和缩进陷阱。

配置文件：

- 主配置：`$XDG_CONFIG_HOME/miyu/config.jsonc`
- 密钥配置：`$XDG_CONFIG_HOME/miyu/secrets.jsonc`
- 自定义系统提示词：`$XDG_CONFIG_HOME/miyu/system-prompt.md`
- 用户 skills：`$XDG_CONFIG_HOME/miyu/skills/`

密钥策略：

- 优先支持环境变量引用，例如 `$env:OPENAI_API_KEY`。
- 允许保存到 `secrets.jsonc`，Unix 下创建时检查权限为 `0600`。
- 不建议默认把 API key 明文写入主配置。

配置工具：

- `miyu config` 是配置入口。
- `miyu config` 第一版就提供类似 `rime-llm-translator` 的 TUI 配置界面。
- `miyu init` 只负责创建默认文件，不和配置界面重复。
- 第一版不提供 `miyu config provider ...` 这类 Provider CLI 子命令，避免入口过散。
- 第一版移除 `miyu tui` 预留命令，避免空命令占用主入口语义。

## XDG 路径

默认路径遵循 XDG Base Directory。

- 配置：`$XDG_CONFIG_HOME/miyu/`
- Skills：`$XDG_CONFIG_HOME/miyu/skills/`
- 数据：`$XDG_DATA_HOME/miyu/`
- 缓存：`$XDG_CACHE_HOME/miyu/`
- 状态：`$XDG_STATE_HOME/miyu/`
- 图片：`$XDG_PICTURES_DIR/miyu/<功能名>/`

常见展开：

- `~/.config/miyu/config.jsonc`
- `~/.config/miyu/secrets.jsonc`
- `~/.config/miyu/skills/`
- `~/.local/share/miyu/`
- `~/.cache/miyu/`
- `~/.local/state/miyu/`
- `~/Pictures/miyu/search-image/`
- `~/Pictures/miyu/image-generation/`
- `~/Pictures/miyu/ocr/`

## 状态文件

参考 tash 的轻量状态模型，Miyu 第一版预留以下状态文件：

- `conversation.jsonl`：当前会话历史，追加写入，assistant 记录可包含 `reasoning` 字段。
- `conversation-YYYYMMDD-HHMMSS.jsonl`：`miyu new` 归档出的旧会话历史。
- `usage.json`：token 和调用量统计。
- `miyu.log`：运行日志。
- `profile.md`：用户环境和偏好画像，后续由 Miyu 生成或更新。
- `memory.jsonl` 或 `memory.db`：长期记忆，后续确定格式。

状态文件默认放在 `$XDG_STATE_HOME/miyu/`。持久知识库、索引和插件数据放在 `$XDG_DATA_HOME/miyu/`。

会话上下文默认复用当前 `conversation.jsonl`。普通 `miyu ...` 和 REPL 都属于同一个当前会话，直到用户运行 `miyu new` 归档旧会话或 `miyu reset` 清空当前会话。

上下文裁剪策略：

- 最大上下文窗口优先从当前 Provider 的 `model_context_chars[default_model]` 读取。
- 未配置模型专属窗口时使用 `context.default_max_chars` 作为 fallback。
- 当前会话历史超过窗口的 `context.trim_at_ratio`，每次请求前会丢弃最旧的若干条消息，目标是释放约 `context.trim_batch_ratio` 的窗口。
- 第一阶段用字符数近似上下文预算，后续可按模型 tokenizer 或 provider 能力替换为 token 估算。

## Agent 响应类型

第一版实现普通 `chat` 回复，并在终端输出层区分 thinking 和正文。

终端输出规则：

- 正文按 Markdown 渲染到终端。
- 流式输出回退为稳定 stdout：thinking 灰色实时输出，正文原样实时输出。
- 主聊天路径暂不做正文 Markdown 渲染，避免 viewport/reflow 破坏普通终端体验。
- `miyu history` 仍可对已完成正文做 Markdown 渲染。

### 插件和 Skills

- Miyu 自带插件直接编码进 Rust 二进制，不从外部插件文件加载。
- AstrBot 插件迁移时只保留 CLI 有意义的能力，去掉 AstrBot、NapCat、OneBot、QQ群、群管理等特供上下文。
- 已迁移内置玄学工具：`draw_zhouyi_hexagram`、`draw_tarot_card`、`draw_fortune_lot`。
- Skills 只读取 `$XDG_CONFIG_HOME/miyu/skills`，不读取 `~/.opencode/skills`。
- Skills 的外部命令执行由 `skills.allow_command_execution` 全局开关控制，默认关闭。
- 默认 Provider 使用 opencode Zen：`https://opencode.ai/zen/v1`，Provider ID 为 `opencode`，默认对话模型 `big-pickle`，默认识图模型 `mimo-v2.5-free`；未配置 API key 时自动使用 `public` 凭证，仅保证官方公开模型可用。默认模型集中定义在 `src/default_models.rs`。
- thinking 默认显示，但用弱化颜色和正文分隔。
- shell intercept 默认只输出正文，不显示 thinking。
- OpenAI-compatible 响应优先读取 `reasoning_content` 或 `reasoning` 字段；如果正文里包含 `<think>...</think>` 或 `<thinking>...</thinking>`，则分离为 thinking。

内部预留响应类型：

- `chat`：自然语言回答。
- `command_suggestion`：命令建议，不自动执行。
- `correction`：拼错命令纠正。
- `tool_call`：工具调用。
- `plan`：多步计划。
- `memory`：记忆读写。
- `ignore`：不处理，交回 shell 或退出。

Shell 拦截入口优先使用结构化响应，避免把危险命令当普通文本处理。

## Skills 系统

Skills 是用户可配置、可覆盖的能力说明或行为模块。

设计原则：

- Skills 放在 `$XDG_CONFIG_HOME/miyu/skills/`。
- Skills 文件使用 Markdown。
- Skills 不等同于 Rust 插件，不直接执行任意代码。
- Skills 可以影响提示词、工具选择、工作流和上下文注入。
- 内置 skills 可以编译进二进制，用户配置可覆盖同名 skill。

内置提示词覆盖策略：

- 如果没有用户提示词，使用二进制内置的 Miyu 提示词。
- 如果配置了用户提示词，完全使用用户提示词覆盖内置提示词。
- 第一版不做内置提示词追加或前置合并。

第一版内置提示词来源：

- 来源文件：`/home/shorin/Documents/miyu_prompt_origin.md`。
- 迁入时保留通用人格、回复风格、终端助手规则、Linux 相关偏好、特殊用词和通用任务边界。
- 迁入时移除 QQ 特化内容，包括 QQ 号、群友称呼、群规、禁言、踢人、撤回、群管理和冒充 QQ 身份判定。
- 插件工具提示词、深度研究提示词、知识库提示词不混入主提示词，后续插件阶段单独迁移。

## 插件系统

插件是 Rust 内置模块，通过 trait 注册到工具注册表。

候选接口：

```rust
trait ToolPlugin {
    fn id(&self) -> &'static str;
    fn tools(&self) -> Vec<ToolSpec>;
}
```

设计原则：

- 第一阶段不支持运行时加载动态库，避免复杂性和安全风险。
- 内置插件默认编译进二进制。
- 用户可以通过配置启用、禁用、覆盖参数或覆盖提示词说明。
- `miyu config` 主菜单提供 `插件配置`，用于启用/禁用内置插件并编辑插件参数。
- 插件数据放在 `$XDG_DATA_HOME/miyu/plugins/<plugin-id>/`。
- 插件缓存放在 `$XDG_CACHE_HOME/miyu/plugins/<plugin-id>/`。

## 网络搜索与 WebFetch

网络搜索应保留 AstrBot 插件里的多后端思路：

- 优先 API 池：Tavily、Firecrawl、AnySearch。
- 如果配置了 API key，优先使用对应 API。
- API 未配置或失败时，先 fallback 到配置的 SearXNG 实例。
- SearXNG 未配置或失败时，最后 fallback 到内置纯 HTTP/HTML 脚本式搜索。
- 保留脚本式搜索 fallback，但不把 Python 作为默认运行时依赖。

WebFetch 可以参考 opencode 的网页读取体验：

- 获取 HTML。
- 提取正文。
- 转 Markdown 或纯文本。
- 限制页面大小和超时。
- 做缓存和 MIME 检查。

## 深度研究

- 原 AstrBot `深度思考` 在 Miyu CLI 中改名为 `deep-research` / `深度研究`。
- 深度研究结果不保存到数据库，最终 Markdown 文件默认写入 XDG Documents 下的 `Miyu/deep-thinking/topic-YYYYMMDD-HHMMSS.md`。
- 运行态任务、轻量索引和 token/usage 记录可以放在 `$XDG_CACHE_HOME/miyu/plugins/deep-research/` 或 `$XDG_STATE_HOME/miyu/usage.json`。
- 完成后工具返回文件路径，让用户自行打开阅读。
- 不保留 AstrBot 的群聊、会话、发起人白名单等控制逻辑。
- 第一版实现同步 `deep_research` 工具：沉思者工具循环、审视者 JSON 审查、R/K/W 引用注册、最终参考资料章节和 Markdown 文件输出。
- 第一版暂不实现后台任务、Dashboard、SQLite 记录库、语义索引、任务取消/修改和文件发送。
- 深度研究沿用挡位设计：`minimal`、`low`、`medium`、`high`、`xhigh`；配置里的审视修正次数和每轮工具步数为 `0` 时按挡位默认，只有 `xhigh` 表示插件层尽量不限制。
- 运行中输出阶段级进度和沉思者内部工具调用状态；不输出工具返回正文。报告写入时进度行可显示完整路径，同时工具 JSON 返回 `archive_path`，由主 AI 决定是否告知用户。

## 图片相关能力

所有图片相关输出默认保存到 XDG 图片目录：

```text
$XDG_PICTURES_DIR/miyu/<功能名>/
```

需要预留终端图片预览：

- Kitty graphics protocol。
- Sixel。
- iTerm2 inline image。
- 不支持时打印文件路径。

识图第一版实现 `vision_analyze` 工具：支持本地图片路径和 http(s) 图片 URL，复用当前 Provider；如果配置了 `plugins.vision.vision_provider_id` 和 `vision_model`，则使用该专用识图 Provider/模型。`print_image` 是默认工具，由“打印图片”插件配置控制；它使用 `chafa` 直接把本地图片打印到当前终端输出，支持 `size` 或 `width`/`height` 控制尺寸；未显式指定尺寸时按 `plugins.print_image.width_percent` / `plugins.print_image.height_percent` 换算当前终端尺寸，`plugins.print_image.enabled=false` 时不注册该工具。

生图第一版实现 `generate_image` 工具，只支持 OpenAI 和 RightCode 两类兼容 API。配置项位于 `plugins.image_generation`：`provider_type=openai|rightcode`、`base_url`、多 API key、`model`、默认宽高比、默认分辨率、输出目录、超时秒数，以及完成后是否自动用 `print_image` 打印。OpenAI 路径调用 `/v1/images/generations`，支持返回 `b64_json` 或 `url`；RightCode 路径复用同一接口，并额外发送 `aspect_ratio` 和更精确的像素 `size`。`generate_image` 返回会带 `printed` 和模型指令：未明确要求展示时不应二次调用 `print_image`，已自动打印时不应重复打印同一路径。

生图输出目录默认读取 XDG Pictures 目录并写入 `Miyu/generated-images`，不硬编码 home 路径。

知识库第一版实现本地文件型知识库：数据默认放 `$XDG_DATA_HOME/miyu/knowledge-base/`，文件放 `files/`，元数据放 `kb_meta.db`，语义侧车索引放 `semantic_index.db`。管理命令为 `miyu kb add/list/search/find/read/remove/reindex/stats/embed reindex`；LLM 工具为 `search_knowledge_base`、`search_knowledge_base_by_name`、`read_knowledge_base_file`，并可选启用 `upload_text_to_knowledge_base`。搜索先跑关键词/文件名/同窗片段；关键词弱命中且 embedding 已配置时，用 OpenAI-compatible `/v1/embeddings` 建立和查询语义 chunk，最终仍返回原文片段。`kb add` 和 AI 上传后会尝试 disown 后台 embedding reindex，使用 lock 文件避免重复运行；未配置 embedding provider/model 时自动跳过。

全局显示配置放在 `display`：`show_reasoning` 控制普通聊天和 REPL 是否显示 thinking/reasoning，shell intercept/plain 模式仍强制隐藏；`show_tool_details` 控制是否显示完整工具参数和工具输出，关闭时只显示工具名和 ok/error。

## 参考项目：tash

`tash` 是一个轻量 fish shell AI middleware。它的定位和 Miyu 的 shell 入口体验很接近。

适合借鉴：

- `query <input>` 的自然语言入口。
- command-not-found 拦截。
- 单二进制、最小依赖、低启动成本。
- `conversation.jsonl`、`usage.json`、`profile.md`、日志文件等状态模型。
- TUI 输出走 stderr、命令结果走 stdout 的分离思路。
- Zellij screen capture 作为终端上下文来源。
- AI 响应类型区分：聊天、命令、历史、记忆、屏幕、计划。

不建议照搬：

- tash 是 Go 项目，Miyu 目标是 Rust。
- tash 强绑定 fish middleware，Miyu 应先做通用 CLI。
- tash 使用 YAML，Miyu 使用 JSONC。
- tash 不是插件化 agent，Miyu 需要从一开始预留工具注册表。

## 第一版验收标准

第一版实现完成时，至少应满足以下行为：

- 仓库根目录存在可构建 Rust 工程，`cargo build` 通过。
- `miyu init` 会创建配置、状态、缓存、数据、skills 目录和默认 `config.jsonc`。
- `miyu config validate` 能读取 JSONC，配置合法时成功，非法时给出清晰错误。
- `miyu config` 能以 TUI 方式创建或编辑至少一个 OpenAI-compatible Provider。
- `miyu paths` 能打印所有 XDG 实际路径。
- `miyu 你好` 会把 `你好` 作为用户消息发送给当前激活 Provider。
- `miyu 帮我 写一个 rust cli` 会把所有位置参数拼成一条消息。
- `miyu` 无参数进入 REPL，可连续输入消息，输入 `exit` 或 `quit` 退出。
- `miyu ask ...` 作为 `miyu ...` 的兼容别名。
- `miyu fish-init` 创建 `~/.config/fish/conf.d/miyu.fish`。
- `miyu history` 能渲染当前会话历史。
- `miyu new` 能归档当前会话并开始新会话。
- `miyu reset` 能清空当前会话历史。
- `miyu --shell-intercept --shell fish -- 你好` 走 shell 拦截专用流程并返回文本。
- `conversation.jsonl` 和 `usage.json` 至少完成基础写入或预留文件创建，assistant 历史可保存 reasoning。
- 不实现具体插件，但存在工具注册表和插件 trait 的最小骨架。

## Compact 后实现摘要

如果上下文被压缩，后续实现者应优先读取本节和全文未定问题。

Miyu 要在 `/home/shorin/Documents/github/Miyu/` 中实现为 Rust 单二进制 CLI AI Agent。当前仓库只有 `README.md` 和 `knowledge-base/`，应在根目录新增 `Cargo.toml`、`src/` 等 Rust 工程文件，不移动 `knowledge-base/`。主配置格式是 JSONC，路径遵循 XDG：`~/.config/miyu/config.jsonc`、`~/.config/miyu/secrets.jsonc`、`~/.config/miyu/skills/*.md`、`~/.local/share/miyu/`、`~/.cache/miyu/`、`~/.local/state/miyu/`、`~/Pictures/miyu/<功能名>/`。

CLI 语义必须是：`miyu [MESSAGE...]` 为主入口，所有位置参数用空格拼成一条用户消息；`miyu` 无参数进入 REPL；`miyu ask [MESSAGE...]` 只是兼容别名；`miyu config` 第一版就提供 TUI 配置；`miyu init` 只初始化文件；`miyu fish-init` 创建 `~/.config/fish/conf.d/miyu.fish`；`miyu history` 渲染当前会话历史；`miyu new` 归档当前会话并开启新会话；`miyu reset` 清空当前会话；`miyu --shell-intercept --shell fish -- <argv...>` 用于 fish `command not found` 自然语言拦截。第一版不保留 `miyu tui` 空命令。

第一版只实现 OpenAI-compatible chat completions、Provider 配置、路径、状态文件、REPL、当前会话上下文、自动历史裁剪、shell fish hook、终端 Markdown 渲染、thinking 区分和空工具注册表。不迁移知识库、天气、搜索、图片、OCR、深度思考等插件。Skills 文件固定用 Markdown。内置 Miyu 提示词编译进二进制；用户自定义提示词放在 `$XDG_CONFIG_HOME/miyu/system-prompt.md`，非空时完全覆盖内置提示词；没有用户提示词时才使用内置提示词。shell 拦截不做吐槽强度配置，语气由提示词控制。

实现时优先保持小而可运行：先构建 CLI、配置、路径和最小 LLM 调用，再补交互式配置、状态文件和 fish hook。不要为了插件系统提前引入动态加载或复杂运行时插件，第一版只需要 Rust trait 和 registry 骨架。
